   682Ôćĺ### Detailed Findings & Fixes
   683Ôćĺ
   684Ôćĺ#### 1. CRITICAL ÔÇö Stored XSS via File Upload Extension Mismatch
   685Ôćĺ- **Route**: `POST /api/upload`
   686Ôćĺ- **Vulnerability**: The extension validation block (`if (!allowedExtensions.includes(ext))`) contained only a comment but no actual enforcement. An attacker could upload a file claiming MIME type `application/pdf` with filename `evil.html`, and it would be saved as `.html` in the public uploads directory, enabling stored XSS.
   687Ôćĺ- **Fix**: Enforced extension-to-MIME-type matching with fallback to the first allowed extension, and added an explicit blocklist of dangerous extensions (html, htm, svg, js, xml, php, asp, exe, bat, sh, etc.).
   688Ôćĺ
   689Ôćĺ#### 2. HIGH ÔÇö Removed/Flagged Notes Accessible by Direct ID
   690Ôćĺ- **Route**: `GET /api/notes/[id]`
   691Ôćĺ- **Vulnerability**: The GET handler fetched notes by ID without checking `status`. A user could directly access notes with `status: 'removed'` or `status: 'flagged'` by guessing/leaking the ID.
   692Ôćĺ- **Fix**: Added status check after fetching: if note is removed/flagged, return 404 unless user is admin/moderator.
   693Ôćĺ
   694Ôćĺ#### 3. HIGH ÔÇö Status Filter Bypass in Notes List
   695Ôćĺ- **Route**: `GET /api/notes`
   696Ôćĺ- **Vulnerability**: The `status` query parameter defaulted to 'active' but accepted any value. A regular user could pass `?status=removed` or `?status=flagged` to see notes they shouldn't.
   697Ôćĺ- **Fix**: Implemented an allowed-statuses list based on user role. Regular users can only see 'active'; admin/moderator users can also see 'processing', 'flagged', 'removed'.
   698Ôćĺ
   699Ôćĺ#### 4-5. MEDIUM ÔÇö Missing Zod Validation on Note Creation/Update
   700Ôćĺ- **Route**: `POST /api/notes`, `PUT /api/notes/[id]`
   701Ôćĺ- **Vulnerability**: All input fields were destructured from the request body without validation. No length limits, no type constraints, potential for oversized payloads or injection.
   702Ôćĺ- **Fix**: Added `createNoteSchema` and `updateNoteSchema` with Zod, including max lengths (title 200, description 2000, tags max 10 items max 50 chars each, etc.) and proper ZodError catch blocks.
   703Ôćĺ
   704Ôćĺ#### 6. MEDIUM ÔÇö filePath Leaked in Note Detail Response
   705Ôćĺ- **Route**: `GET /api/notes/[id]`
   706Ôćĺ- **Vulnerability**: The `filePath` field (e.g., `/uploads/1234-abc.pdf`) was returned to all users, revealing server file structure.
   707Ôćĺ- **Fix**: Removed `filePath` from the formatted response. Downloads are handled via the dedicated `/api/download/[id]` endpoint.
   708Ôćĺ
   709Ôćĺ#### 7. MEDIUM ÔÇö Flagged Notes Downloadable
   710Ôćĺ- **Route**: `GET /api/download/[id]`
   711Ôćĺ- **Vulnerability**: Only `status === 'removed'` was blocked. Notes with `status === 'flagged'` could still be downloaded.
   712Ôćĺ- **Fix**: Added `note.status === 'flagged'` to the blocking check.
   713Ôćĺ
   714Ôćĺ#### 8. MEDIUM ÔÇö No Zod Validation on Comments
   715Ôćĺ- **Route**: `POST /api/comments`, `PUT /api/comments`
   716Ôćĺ- **Vulnerability**: No input validation, no content length limit. Users could submit arbitrarily long comments.
   717Ôćĺ- **Fix**: Added `createCommentSchema` and `editCommentSchema` with `z.string().min(1).max(2000)` for content, proper ZodError handling.
   718Ôćĺ
   719Ôćĺ#### 9. MEDIUM ÔÇö No Zod Validation on Ratings
   720Ôćĺ- **Route**: `POST /api/ratings`
   721Ôćĺ- **Vulnerability**: Rating value was parsed with `parseInt()` on raw input ÔÇö could be NaN, negative, or extremely large.
   722Ôćĺ- **Fix**: Added `rateNoteSchema` with `z.number().int().min(1).max(5)` for value. Removed manual parseInt/isNaN checks.
   723Ôćĺ
   724Ôćĺ#### 10-11. MEDIUM ÔÇö Admin Role Escalation
   725Ôćĺ- **Route**: `POST /api/admin` (updateUser action)
   726Ôćĺ- **Vulnerability**: The `role` field was `z.string().optional()`, accepting arbitrary strings like "super_admin_hacked". Also, an admin could set their own role to a higher privilege level.
   727Ôćĺ- **Fix**: Changed `role` to `z.enum(VALID_ROLES)` restricting to known role strings. Added check preventing non-super_admin users from modifying their own role.
   728Ôćĺ
   729Ôćĺ#### 12. MEDIUM ÔÇö Email Leak in Admin listUsers
   730Ôćĺ- **Route**: `POST /api/admin` (listUsers action)
   731Ôćĺ- **Vulnerability**: The `listUsers` action returned `email: true` in the select for all admin/moderator users. Moderators should not see user emails. Also, the search `OR` clause included email search for non-super_admin users.
   732Ôćĺ- **Fix**: Made email field conditional on `user.role === 'super_admin'`. Removed email from search `OR` clause for non-super_admin.
   733Ôćĺ
   734Ôćĺ#### 13. MEDIUM ÔÇö Any User Can Create Subjects
   735Ôćĺ- **Route**: `POST /api/subjects`
   736Ôćĺ- **Vulnerability**: Any authenticated user could create subjects, which should be an admin-only operation (similar to colleges).
   737Ôćĺ- **Fix**: Added role check requiring admin/super_admin/moderator.
   738Ôćĺ
   739Ôćĺ#### 14. MEDIUM ÔÇö AI Process GET No Auth
   740Ôćĺ- **Route**: `GET /api/ai/process`
   741Ôćĺ- **Vulnerability**: The GET endpoint for checking AI processing status had no authentication or ownership check. Any user could query any note's processing status.
   742Ôćĺ- **Fix**: Added `getAuthUser()` check and ownership/role validation matching the POST endpoint.
   743Ôćĺ
   744Ôćĺ#### 15. MEDIUM ÔÇö extractedText Visible to Non-Owners
   745Ôćĺ- **Route**: `GET /api/notes/[id]`
   746Ôćĺ- **Vulnerability**: The full `extractedText` (potentially copyrighted content) was returned to all users, not just the owner.
   747Ôćĺ- **Fix**: Made `extractedText` conditional: only returned to the note owner or admin/moderator. Other users see `null`.
