   682→### Detailed Findings & Fixes
   683→
   684→#### 1. CRITICAL — Stored XSS via File Upload Extension Mismatch
   685→- **Route**: `POST /api/upload`
   686→- **Vulnerability**: The extension validation block (`if (!allowedExtensions.includes(ext))`) contained only a comment but no actual enforcement. An attacker could upload a file claiming MIME type `application/pdf` with filename `evil.html`, and it would be saved as `.html` in the public uploads directory, enabling stored XSS.
   687→- **Fix**: Enforced extension-to-MIME-type matching with fallback to the first allowed extension, and added an explicit blocklist of dangerous extensions (html, htm, svg, js, xml, php, asp, exe, bat, sh, etc.).
   688→
   689→#### 2. HIGH — Removed/Flagged Notes Accessible by Direct ID
   690→- **Route**: `GET /api/notes/[id]`
   691→- **Vulnerability**: The GET handler fetched notes by ID without checking `status`. A user could directly access notes with `status: 'removed'` or `status: 'flagged'` by guessing/leaking the ID.
   692→- **Fix**: Added status check after fetching: if note is removed/flagged, return 404 unless user is admin/moderator.
   693→
   694→#### 3. HIGH — Status Filter Bypass in Notes List
   695→- **Route**: `GET /api/notes`
   696→- **Vulnerability**: The `status` query parameter defaulted to 'active' but accepted any value. A regular user could pass `?status=removed` or `?status=flagged` to see notes they shouldn't.
   697→- **Fix**: Implemented an allowed-statuses list based on user role. Regular users can only see 'active'; admin/moderator users can also see 'processing', 'flagged', 'removed'.
   698→
   699→#### 4-5. MEDIUM — Missing Zod Validation on Note Creation/Update
   700→- **Route**: `POST /api/notes`, `PUT /api/notes/[id]`
   701→- **Vulnerability**: All input fields were destructured from the request body without validation. No length limits, no type constraints, potential for oversized payloads or injection.
   702→- **Fix**: Added `createNoteSchema` and `updateNoteSchema` with Zod, including max lengths (title 200, description 2000, tags max 10 items max 50 chars each, etc.) and proper ZodError catch blocks.
   703→
   704→#### 6. MEDIUM — filePath Leaked in Note Detail Response
   705→- **Route**: `GET /api/notes/[id]`
   706→- **Vulnerability**: The `filePath` field (e.g., `/uploads/1234-abc.pdf`) was returned to all users, revealing server file structure.
   707→- **Fix**: Removed `filePath` from the formatted response. Downloads are handled via the dedicated `/api/download/[id]` endpoint.
   708→
   709→#### 7. MEDIUM — Flagged Notes Downloadable
   710→- **Route**: `GET /api/download/[id]`
   711→- **Vulnerability**: Only `status === 'removed'` was blocked. Notes with `status === 'flagged'` could still be downloaded.
   712→- **Fix**: Added `note.status === 'flagged'` to the blocking check.
   713→
   714→#### 8. MEDIUM — No Zod Validation on Comments
   715→- **Route**: `POST /api/comments`, `PUT /api/comments`
   716→- **Vulnerability**: No input validation, no content length limit. Users could submit arbitrarily long comments.
   717→- **Fix**: Added `createCommentSchema` and `editCommentSchema` with `z.string().min(1).max(2000)` for content, proper ZodError handling.
   718→
   719→#### 9. MEDIUM — No Zod Validation on Ratings
   720→- **Route**: `POST /api/ratings`
   721→- **Vulnerability**: Rating value was parsed with `parseInt()` on raw input — could be NaN, negative, or extremely large.
   722→- **Fix**: Added `rateNoteSchema` with `z.number().int().min(1).max(5)` for value. Removed manual parseInt/isNaN checks.
   723→
   724→#### 10-11. MEDIUM — Admin Role Escalation
   725→- **Route**: `POST /api/admin` (updateUser action)
   726→- **Vulnerability**: The `role` field was `z.string().optional()`, accepting arbitrary strings like "super_admin_hacked". Also, an admin could set their own role to a higher privilege level.
   727→- **Fix**: Changed `role` to `z.enum(VALID_ROLES)` restricting to known role strings. Added check preventing non-super_admin users from modifying their own role.
   728→
   729→#### 12. MEDIUM — Email Leak in Admin listUsers
